01
Foundation · 8 minutes

What is GitHub, really?

⏱ 0–8 min

Think of it as Google Drive — but for code

You already know what Google Drive does: it stores your files in the cloud, lets you access them from anywhere, and keeps a history of changes. GitHub does the same thing — but with a superpower: it remembers every single change you ever made, who made it, and why.

Real-world analogy

Imagine you're writing a novel. Every time you make edits, instead of saving over the old version, you take a photograph of every page. You can go back to any photograph at any time. You can also share your novel with a friend, both edit different chapters at the same time, and then combine your changes. That's GitHub.

Two things in one name

📦

Git

The tool that tracks changes to your files. It runs on your computer. Think of it as the "save history" engine.

☁️

GitHub

The website (github.com) that stores your Git history in the cloud. It's where you share, backup, and collaborate. GitHub is to Git what Google Drive is to your hard drive.

🔗

Together

Git tracks changes locally on your computer. GitHub stores those changes online. You use both together — Git does the work, GitHub stores the result.

Why should a non-developer care?

You already used GitHub to deploy your AI learning app. Here's what GitHub actually gives you:

💾

Backup

Your code is safe even if your computer dies. Pull it down on any device in minutes.

⏪

Undo anything

Made a mistake that broke everything? Go back to any previous version instantly.

🌍

Free hosting

GitHub Pages turns your repo into a live website — for free. You already used this.

👥

Collaboration

Share your project with anyone. They can suggest changes without breaking your original.

💡

Key insight: GitHub is not just for developers. Writers, designers, researchers, and project managers all use it. Anything that involves files that change over time benefits from GitHub.

Chapter 1 Summary
  • Git = the save-history engine on your computer
  • GitHub = the website that stores your history in the cloud
  • Together they give you: backup, undo, free hosting, and collaboration

02
Concepts · 10 minutes

The 6 words you need to know

⏱ 8–18 min

GitHub has its own vocabulary. Don't let the words intimidate you — each one has a simple, everyday equivalent. Learn these 6 words and you understand 90% of GitHub.

📁

Repository (Repo)

A folder for your project. It contains all your files AND the full history of every change ever made. Think: project folder with memory.

📸

Commit

A saved snapshot of your files at a point in time. Like pressing save — but it remembers what you changed and why. Think: a photo of your work.

⬆️

Push

Sending your local commits up to GitHub. Your changes go from your computer to the cloud. Think: uploading to Google Drive.

⬇️

Pull

Downloading the latest changes from GitHub to your computer. Think: syncing from Google Drive.

🌿

Branch

A separate copy of your project where you can experiment safely. Your main version stays untouched. Think: a rough draft that doesn't affect the original.

🔀

Merge

Combining changes from one branch back into your main version. Think: accepting tracked changes in Word.

How they connect

Here's the flow from making a change on your computer to it being live on GitHub:

Edit files
on your computer
→
git add
select what to save
→
git commit
take the snapshot
→
git push
send to GitHub
The 3-step flow you'll use every single time
Real-world analogy

Think of it like sending a parcel. git add = packing items into a box. git commit = sealing the box and writing what's inside on the label. git push = handing the parcel to the courier (GitHub). Every step is deliberate — you decide what goes in, you label it, then you send it.

What a commit actually looks like

Every commit stores three things: what changed, a message you wrote explaining why, and a timestamp. This creates a complete history you can always scroll through.

COMMIT HISTORY — what GitHub stores
# GitHub shows you a history like this:

c9127ce  Move hour files into hours folder     ← today, by adtprabhu
d4705a1  Upload correct app files              ← 1 hour ago
9ae54d7  Remove wrong folder                   ← 2 hours ago
3f1b59b  First upload                          ← yesterday

# Each line = one commit = one snapshot in time
# You can go back to ANY of these at any time

Quick check — Chapter 2

Make sure the 6 words are clear before moving on.

Q1. You've made changes to your files and want to save a snapshot. Which command do you use?
✅ Correct! git commit takes the snapshot. Remember: add → commit → push. Commit is step 2 — it seals and labels the box but doesn't send it yet.
Q2. What is a "repository"?
✅ Correct! A repo is a project folder with memory. It stores every file AND the entire history of every change ever made to those files.
Q3. You edited files on your computer and committed them. A colleague also made changes on GitHub. How do you get their changes onto your computer?
✅ Correct! git pull downloads the latest changes from GitHub to your computer. Push = upload. Pull = download. You'll use both constantly.

03
Hands-on · 10 minutes

Setting up your first repository

⏱ 18–28 min

There are two ways to start a repo. Which one you use depends on whether you're starting fresh or if files already exist on your computer.

Way 1: Start on GitHub, then download

Best when starting a brand new project. Create the repo online, then download it.

1

Create on GitHub

Go to github.com → click the + button → New repository. Give it a name, set it to Public, click Create.

2

Clone it to your computer

Click the green Code button → copy the URL. Then in your terminal: git clone [URL]. This downloads the repo as a folder on your computer.

3

Start adding files

Open the downloaded folder. Add your files inside it. Now you're ready to commit and push.

TERMINAL — clone a repo
# Download a repo from GitHub to your computer
git clone https://github.com/yourusername/your-repo.git

# This creates a folder with the same name as your repo
# Go into it:
cd your-repo

# Now you're inside the repo. Any files you add here
# are tracked by Git automatically.

Way 2: Have files already? Turn the folder into a repo

Best when you already have files on your computer and want to add GitHub to them.

TERMINAL — turn existing folder into a repo
# Go into your existing folder
cd my-existing-project

# Tell Git to start tracking this folder
git init

# Connect it to a GitHub repo you created online
git remote add origin https://github.com/yourusername/your-repo.git

# Add all files, commit, and push
git add .
git commit -m "First commit"
git push -u origin main
ℹ️

What does git remote add origin mean? "Remote" = a GitHub repo online. "Origin" = the nickname we give it (it's just a shortcut name). So this command says: "Connect my local folder to this GitHub URL and call it 'origin' for short."

⚠️

The dot in git add . matters! The dot means "add ALL files in this folder." Without the dot you'd have to list every file by name. With the dot, everything gets added at once.

TRY IT

Create a test repo right now

Go to github.com and create a new public repo called test-repo. Don't add any files yet. Then copy the clone URL and run git clone [URL] in your terminal. Did a new folder appear on your computer?

After cloning, a folder named test-repo should appear wherever you ran the command. Run ls to see it. Go inside with cd test-repo. Run ls -la and you'll see a hidden .git folder — that's where Git stores all the history. Never delete or edit that folder.

04
Core skill · 10 minutes

The daily workflow: add → commit → push

⏱ 28–38 min

This is the workflow you'll use every single time you make changes. Once it's muscle memory, using GitHub becomes effortless. Three commands, always in this order.

1

Make your changes

Edit files, add new files, delete things — whatever you need to do in your project folder. Git is watching, but it doesn't act until you tell it to.

2

git add — choose what to save

Tell Git which changed files to include in your next snapshot. git add . adds everything. git add filename.html adds just one file.

3

git commit — take the snapshot

Seal the snapshot with a message describing what you did. This message is for you — write it clearly so future-you understands what happened.

4

git push — send to GitHub

Upload your commits to GitHub. Now they're backed up online and your live site (if using GitHub Pages) will update automatically.

TERMINAL — the daily workflow
# Step 1: See what's changed (always good to check first)
git status
On branch main
Changes not staged for commit:
  modified: index.html
  modified: hours/hour-1.html

# Step 2: Add everything
git add .

# Step 3: Commit with a clear message
git commit -m "Fix typo in hour 1 lesson"
[main a3f2c1b] Fix typo in hour 1 lesson
 2 files changed, 4 insertions(+), 2 deletions(-)

# Step 4: Push to GitHub
git push
To https://github.com/adtprabhu/ai-learning-app.git
   c9127ce..a3f2c1b  main -> main

# Done! Your changes are live on GitHub.

Writing good commit messages

A commit message is a note to your future self. Here's the difference between good and bad messages:

❌

Bad commit messages

"stuff"
"fix"
"changes"
"asdfgh"
"update 2 final FINAL"

✅

Good commit messages

"Fix navigation links on mobile"
"Add quiz to hour 3 lesson"
"Remove wrong folder from repo"
"Update homepage headline text"

💡

Pro tip: Write commit messages as if completing the sentence "This commit will..." — "Fix navigation links on mobile" = "This commit will fix navigation links on mobile." It forces clarity.

git status — your best friend

Before doing anything, run git status. It tells you exactly what's happening: which files changed, what's been added, what hasn't been committed yet. When something goes wrong, run git status first — it almost always tells you what to do next.

TRY IT

Complete the full workflow

In your test-repo folder, create a new file called notes.txt and write anything inside it. Then run the full workflow: git add . → git commit -m "Add notes file" → git push. Then go to github.com and check your repo — does the file appear?

After pushing, refresh your GitHub repo page. You should see notes.txt listed alongside your README. Click it to see the contents. Also notice the commit message "Add notes file" appears next to the file — GitHub shows which commit last touched each file. This is the history at work.

05
Key concept · 8 minutes

Branches — experiment without fear

⏱ 38–46 min

Branches are one of GitHub's most powerful features. They let you try new things — new designs, new features, risky changes — without touching your working version.

Real-world analogy

Think of branches like making a photocopy of a document before you start marking it up with red pen. The original is safe. You do your experiments on the copy. If you like the result, you replace the original. If you hate it, you throw the copy away.

Main branch vs feature branches

🏠

main branch

Your official, live version. This is what GitHub Pages serves to the world. Always keep this clean and working.

🧪

Feature branches

Separate copies where you experiment. Named things like "add-dark-mode" or "redesign-header". If it works, merge it. If not, delete it — main is untouched.

TERMINAL — working with branches
# See all branches (star = the one you're on)
git branch
* main

# Create a new branch and switch to it
git checkout -b redesign-homepage
Switched to a new branch 'redesign-homepage'

# Now make changes, add, commit as normal
# These commits ONLY exist on this branch
git add .
git commit -m "Try new homepage layout"

# Switch back to main (your original is untouched)
git checkout main

# Happy with the redesign? Merge it into main
git merge redesign-homepage

# Done with the branch? Delete it
git branch -d redesign-homepage

The visual flow

main ●
→
commit A
→
commit B
→
...
↓ create branch here
feature ●
→
try X
→
try Y
→
merge ✓
main stays safe while you experiment on the branch
💡

For solo projects (like your AI learning app), you probably won't use branches much — working directly on main is fine. Branches become essential when collaborating with others or when you want to try risky changes without fear.

TRY IT

Create and merge a branch

In your test repo, create a branch called experiment. Add a file called experiment.txt, commit it. Switch back to main — notice the file disappears! Now merge the branch. Does the file reappear?

When you switch back to main with git checkout main, experiment.txt will vanish — it only exists on the experiment branch. After git merge experiment, it reappears on main. This is branches working perfectly: your main was safe the whole time, and you brought in only what you wanted.

06
Troubleshooting · 8 minutes

When things go wrong — common fixes

⏱ 46–54 min

Every GitHub user hits these problems. None of them are scary once you know the fix. Here are the 6 most common errors you'll encounter and exactly how to solve them.

Problem 1: "Please tell me who you are"

Git doesn't know your name yet. This happens the first time you use Git on a computer.

FIX
git config --global user.email "you@example.com"
git config --global user.name "Your Name"
# Run these once and Git remembers forever

Problem 2: "Authentication failed" when pushing

You used your GitHub password instead of a Personal Access Token. GitHub stopped accepting passwords in 2021.

⚠️

When GitHub asks for a password in the terminal, it never means your account password. It always means a Personal Access Token (a long code starting with ghp_). Generate one at: github.com → Settings → Developer settings → Personal access tokens.

Problem 3: "Updates were rejected — fetch first"

GitHub has changes your computer doesn't have yet. You need to download them before you can upload yours.

FIX
# Download the latest changes first
git pull
# Then push your changes
git push

Problem 4: Committed the wrong thing

You committed something by mistake and haven't pushed yet. Undo the commit but keep your file changes:

FIX — undo last commit, keep files
# Undo the last commit but keep your file changes
git reset --soft HEAD~1
# Your files are still there, commit is gone
# Now you can re-commit correctly

Problem 5: My GitHub Pages site shows a 404

🔍

Check 1: index.html location

index.html must be at the ROOT of your repo — not inside a subfolder. If it's in a folder, the site 404s.

⏳

Check 2: Wait and refresh

GitHub Pages takes 1–3 minutes to build after a push. Always wait before assuming it's broken.

⚙️

Check 3: Pages settings

Settings → Pages → Branch must be set to main. If it says "None", GitHub Pages isn't activated.

Problem 6: "No such file or directory"

You're in the wrong folder. Always check where you are before running Git commands.

FIX
# See where you currently are
pwd
/Users/adtprabhu/Downloads

# List what's in this folder
ls
ai-learning-app   Documents   Downloads

# Navigate into your repo
cd ai-learning-app

# Now run your git commands
💡

Golden rule: When anything goes wrong, run git status first. It shows exactly what state your repo is in and often suggests the exact command you need to fix it.

Your permanent cheat sheet

⏱ 54–60 min

Bookmark this page. Every command you'll ever need as a non-developer using GitHub, in plain English.

📋 GitHub Commands — Plain English Edition

git status
Show what's changed. Run this first when confused.
git add .
Stage all changed files to be saved in the next commit.
git commit -m "message"
Take a snapshot of staged files with a description.
git push
Upload your commits to GitHub. Updates your live site.
git pull
Download the latest changes from GitHub to your computer.
git clone [URL]
Download a GitHub repo to your computer for the first time.
git log
See the full history of all commits. Press Q to exit.
git diff
Show exactly what changed in your files since last commit.
git checkout -b name
Create a new branch and switch to it immediately.
git checkout main
Switch back to your main branch.
git merge branchname
Bring changes from a branch into your current branch.

Undo the last commit but keep your file changes.

Delete a folder from the repo (then commit + push).
pwd
Show which folder you're currently in.
ls
List all files and folders in the current directory.
cd foldername
Go into a folder. cd .. goes up one level.

Final quiz — the whole course

Let's make sure everything has landed.

Q1. You just edited 3 files and want to save them to GitHub. What is the correct order of commands?
✅ Always: add → commit → push. Add stages the files, commit snapshots them, push sends them to GitHub.
Q2. You want to try a risky redesign without breaking your live site. What should you do?
✅ Branches exist exactly for this. Create a branch, experiment safely, merge if happy. Main stays untouched the whole time.
Q3. git push fails with "authentication failed". What's the most likely cause?
✅ GitHub requires a Personal Access Token (ghp_...) as the password in the terminal — not your account password. Generate one at github.com → Settings → Developer settings → Personal access tokens.
Q4. What is the difference between git and GitHub?
✅ Git = local tool that tracks changes. GitHub = website that stores your Git history in the cloud. You use both together. Git does the work, GitHub stores the result.
