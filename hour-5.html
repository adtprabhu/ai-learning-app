Part 1 — Concepts (0–15 min)
\n

Plan first, build second

\n

The most common reason beginner projects fail isn\'t bad code — it\'s starting to build without knowing what you\'re building. In this hour we define the entire project, so every subsequent hour has a clear target.

\n
Real-world analogy

An architect doesn\'t start laying bricks on day one. They draw blueprints, understand what the building needs to do, who will use it, and how it all connects. Then they build. We\'re drawing blueprints today.

\n

What we\'re building: Smart Study Helper

\n
🎯

The problem

Students struggle to test themselves after studying. Reading notes is passive — quizzing yourself is active and far more effective for memory retention (this is called the \'testing effect\').

👤

The user

A student aged 12–18 who wants help studying any school subject. They may be studying alone, without a tutor, and need immediate feedback and explanation.

💡

The solution

An AI-powered app where students type a topic, get a quiz, answer it, receive instant feedback and explanations, hear answers spoken aloud, and track what they\'ve mastered.

📱

The platform

A web app — works on any device via browser. No app store, no installation, just a URL to share.

\n

The 7 features — one per hour block

\n
1

Quiz Generator (Hour 6)

Claude creates 5 multiple-choice questions on any topic the student types.

2

Image Input (Hour 7)

Student uploads a photo of their notes/textbook — Claude explains and quizzes them on it.

3

Live Search (Hour 8)

Google Search API keeps study content current and accurate.

4

Document Upload (Hour 9)

Student uploads their actual PDF textbook — Claude answers questions from it.

5

Progress Tracking (Hour 10)

Firebase stores scores and history. Claude analyses patterns to identify weak spots.

6

Voice Mode (Hour 12)

Google TTS reads answers aloud. Google STT lets students speak questions.

7

Multi-language (Hour 13)

Google Translate makes the whole app work in the student\'s own language.

Exercise 2API cost estimate
\n
\n

Estimate the token cost of one study session. Assume: system prompt = 200 tokens, each question generation = 300 tokens, each answer grading = 150 tokens, 5 questions per session. How many total input tokens per session? At roughly $3 per million input tokens for Claude Sonnet, what does one session cost?

\n \n \n
System prompt per call (sent 10 times — 5 generates + 5 grades) = 200 × 10 = 2,000 tokens. Question prompts: 300 × 5 = 1,500. Grading prompts: 150 × 5 = 750. Total input ≈ 4,250 tokens. At $3/million tokens = $0.013 per session — about 1.3 cents. Very affordable, even at scale.

Part 3 — Review (45–60 min)
\n

Check your understanding

Answer all questions to see your score.

Q1. Why is defining the user journey before writing code so important?
✅ Correct! The user journey reveals your actual technical requirements. Every step the user takes maps to code, API calls, and data. Without this map, you build the wrong things and waste time.
Q2. In the Smart Study Helper, which of these is NOT an AI touchpoint?
✅ Correct! Storing data in Firebase is a database operation — no AI involved. Firebase just saves and retrieves data. The AI touchpoints are all about understanding, generating, and reasoning with text.
Q3. A student session uses roughly 4,250 input tokens. Your app gets 1,000 students per day. What is your approximate monthly API cost (Claude Sonnet at $3/M input tokens)?
✅ Correct! 4,250 × 1,000 × 30 = 127,500,000 tokens/month. At $3/M = $382. But this is just input tokens. Add roughly the same for output tokens = ~$765/month at 1,000 daily users. This is why caching (Hour 17) is important.
