Part 1 — Concepts (0–15 min)
\n

AI memory: a whiteboard, not a notebook

\n

Here\'s one of the most important things to understand before building AI apps: AI does not remember you. Every conversation starts completely fresh. There\'s no database of past chats the AI searches through. It only knows what\'s currently in its context window.

\n
Real-world analogy

Imagine a brilliant doctor who has total amnesia between appointments. Every time you visit, you must re-explain your medical history from scratch. But while you\'re in the room with them, they remember everything you\'ve said in this visit perfectly. That\'s exactly how AI context works.

\n

What is a context window?

\n
📋

Context window

The total amount of text the AI can \'see\' at one time. This includes your system prompt, the full conversation history, and the current message. Think of it as a whiteboard — everything on it is visible; anything erased is gone.

📏

Measured in tokens

Context limits are measured in tokens (roughly 1 token ≈ 0.75 words). Claude\'s context window is very large — hundreds of thousands of tokens — but it\'s not infinite.

🕐

Recency matters

When the context gets very full, older parts of the conversation may get less \'attention\' from the model. The most recent messages are always most influential.

💸

Cost = tokens

Every token in your context costs money via the API. A huge system prompt + long conversation history = higher cost. Good apps manage context carefully.

\n

How conversation history works in apps

\n

When you build an app with the Claude API, you\'re responsible for sending the conversation history yourself. The API doesn\'t store it — you do.

\n
\n
javascript
\n 
// Each API call gets the full conversation history\nconst messages = [\n  { role: "user",      content: "What is photosynthesis?" },\n  { role: "assistant", content: "Photosynthesis is how plants make food using sunlight..." },\n  { role: "user",      content: "Can you explain it more simply?" }\n  // ↑ You append each new message, then send the whole array\n];\n\nconst response = await fetch("https://api.anthropic.com/v1/messages", {\n  method: "POST",\n  headers: { "Content-Type": "application/json" },\n  body: JSON.stringify({\n    model: "claude-sonnet-4-20250514",\n    max_tokens: 1000,\n    system: "You are a helpful science tutor.",\n    messages: messages  // ← the full history every single time\n  })\n})
\n
\n
⚠️

If you start a new API call without including the conversation history, Claude has no idea what was said before. Your app must store and re-send the history every time. This is called \'stateless\' API design — the server remembers nothing.

Part 2 — Hands-on Practice (15–45 min)
\n

Feel the context window in action

\n
1

Test memory limits

In Claude, have a long conversation about a topic. Then, in a completely new chat window, ask about that topic. Notice: Claude remembers nothing. This is intentional — each conversation is a fresh slate.

2

Test within-conversation memory

In one conversation, tell Claude: \'My name is Alex, I am 14, I love football, I find maths hard.\' Then ask 10 unrelated questions. After that, ask: \'What do you know about me?\' It will remember everything from this conversation.

3

Design your context strategy

Think about your Study Helper app: what does Claude always need to know? (Student name, grade, subject, difficulty level?) This goes in the system prompt. What changes per message? (The question the student asks.) This is the user prompt.

\n
\n
Exercise 1Context management design
\n
\n

For your Study Helper app, write a list of: (1) information that should always be in the system prompt (static context), and (2) information that should be in the conversation history (dynamic context), and (3) information you might need to summarise when the conversation gets long.

\n \n \n
Static (system prompt): app purpose, AI persona, tone, topic scope, output format rules. Dynamic (conversation): student\'s questions, Claude\'s answers, which topics have been covered. Summarise when long: compress earlier exchanges into a brief summary like \'We covered: mitosis basics (student understood well), cell division stages (student struggled with anaphase).\' This keeps the context focused without losing key info.

Exercise 2Build a simple memory store
\n
\n

In a text file or on paper, design the data structure that your app will use to store conversation history. What fields does each message need? (Hint: look at the API format above.)

\n \n \n
Each message needs: role (either \'user\' or \'assistant\'), content (the text). Optionally: timestamp, topic_tag (for filtering), token_count (for managing limits). Your app will maintain an array of these objects and send the full array with every API call.

Part 3 — Review (45–60 min)
\n

Check your understanding

Answer all questions to see your score.

Q1. You ask Claude in a new chat \'What did we talk about yesterday?\' What will Claude say?
✅ Correct! Claude has no persistent memory between separate conversations. Each new chat is completely fresh. Only within a single conversation does it remember what was said.
Q2. In your app, a user has a 50-message conversation with Claude. For the 51st message, what must your code do?
✅ Correct! The API is stateless. You must send the complete conversation history every single time. Your app stores and manages this history — the API knows nothing between calls.
Q3. What is the main trade-off of having a very long system prompt?
✅ Correct! Every token in your system prompt is sent with every API call and costs money. A 2,000-token system prompt adds 2,000 tokens of cost to every single message your app sends. Keep system prompts tight and purposeful.
