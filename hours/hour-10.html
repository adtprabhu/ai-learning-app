Storing data with Google

AI memory lasts one session. Firebase gives your app real memory — scores, history, preferences — persisted across days and devices. Learn to set up Firestore and wire it to your Study Helper.

☁️

Firebase Firestore

A cloud database from Google. NoSQL, real-time, free tier is generous. No server to manage — your app talks directly to it via a JavaScript SDK.

📁

Data structure

Firestore stores documents in collections. Think: folder (collection) > file (document) > content (fields). Example: users > userId > {name, grade, totalSessions}

🔄

Real-time updates

Firestore can push updates to your app the instant data changes. Useful for showing live progress bars or syncing across devices.

🔒

Security rules

Firestore security rules control who can read/write what. Always set rules — the default 'allow all' is dangerous in production.

// Firebase setup — add to your HTML :
// 
// 

const firebaseConfig = {
  apiKey: "your-api-key",        // from Firebase console
  authDomain: "your-app.firebaseapp.com",
  projectId: "your-project-id"
};

firebase.initializeApp(firebaseConfig);
const db = firebase.firestore();

// Save a quiz session result
async function saveQuizResult(userId, topic, score, totalQuestions, wrongTopics) {
  await db.collection('sessions').add({
    userId,
    topic,
    score,
    totalQuestions,
    percentage: Math.round((score / totalQuestions) * 100),
    wrongTopics,        // array of topics the student got wrong
    timestamp: firebase.firestore.FieldValue.serverTimestamp()
  });
}

// Load a student's history
async function getStudentHistory(userId) {
  const snapshot = await db.collection('sessions')
    .where('userId', '==', userId)
    .orderBy('timestamp', 'desc')
    .limit(20)
    .get();

  return snapshot.docs.map(doc => ({ id: doc.id, ...doc.data() }));
}

// Let Claude analyse the history for insights
async function getAIInsights(userId) {
  const history = await getStudentHistory(userId);

  const summary = history.map(s =>
    `${s.topic}: ${s.percentage}% (wrong: ${s.wrongTopics?.join(', ') || 'none'})`
  ).join('\n');

  const prompt = `A student's recent quiz history:
${summary}

Identify: (1) their strongest topic, (2) their weakest topic, (3) one specific recommendation.
Respond as JSON: {"strongest": "...", "weakest": "...", "recommendation": "..."}`;

  const response = await fetch("https://api.anthropic.com/v1/messages", {
    method: "POST",
    headers: { "Content-Type": "application/json", "x-api-key": process.env.CLAUDE_API_KEY, "anthropic-version": "2023-06-01" },
    body: JSON.stringify({ model: "claude-sonnet-4-20250514", max_tokens: 300, messages: [{ role: "user", content: prompt }] })
  });

  const data = await response.json();
  return JSON.parse(data.content[0].text.trim());
}

Part 3 — Review (45–60 min)

Check your understanding

Answer all questions to see your score.

Q1. What type of database is Firebase Firestore?
✅ Firestore is NoSQL — it stores flexible JSON-like documents rather than rigid tables with fixed columns. This suits AI app data well because the structure can vary (some sessions have wrong topics, some don't).
Q2. Why should you NEVER use Firestore's default 'allow all' security rules in production?
✅ The default rule opens your entire database to the public. Set rules like: allow read, write: if request.auth.uid == resource.data.userId; — users can only access their own data.
Q3. A student's quiz history is 50 sessions long. Why do we add .limit(20) when querying?
✅ Each Firestore document read costs a tiny amount. For AI insights, 20 recent sessions give Claude more than enough data. Loading all 50 unnecessarily triples the cost and slows the query.
