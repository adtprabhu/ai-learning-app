Part 1 — Concepts (0–15 min)

Building the real UI

Great AI behind a confusing interface feels broken. Build a clean, mobile-friendly UI with proper loading states, smooth feedback, and a layout students actually want to use.

📱

Mobile-first

Most students use phones. Design for a 375px wide screen first. Scale up for desktop. Use CSS Grid and flexbox, not fixed widths.

⏳

Loading states

Every AI call takes 1–3 seconds. Show a skeleton or spinner immediately. Never let the UI appear frozen — users assume it's broken.

🎯

Progressive disclosure

Don't show everything at once. Show the topic input first. After submit, show questions one at a time. After completion, show the score. Each step reveals the next.

♿

Accessibility basics

Add aria-label to icon buttons. Ensure colour is not the only indicator (add text too). Use semantic HTML (button, nav, main, section). Test keyboard navigation.

Exercise 2Test on a real phone

Open the app on your phone browser (use ngrok or VS Code Live Share to expose localhost). What breaks? Is the text too small? Are buttons too small to tap? Is the loading state visible?

Common phone issues: touch targets under 44px (too small to tap reliably), text under 14px (too small to read), inputs that zoom the page on focus (fix: font-size: 16px on inputs). Test with slow 3G in Chrome DevTools Network tab — does the loading state appear quickly enough?

Q1. What is 'progressive disclosure' in UI design?
✅ Progressive disclosure reduces cognitive load. Don't show the quiz questions, score screen, and history all at once — show the topic input first, then questions after submit, then score after completion. Each step feels natural and focused.
Q2. Why must input elements have font-size: 16px on mobile?
✅ iOS Safari automatically zooms the viewport when focusing an input with font-size < 16px — to make it readable. This breaks your layout. Set font-size: 16px on all inputs to prevent the zoom. This is one of the most common mobile web bugs.
Q3. Your skeleton loading screen shows immediately, but the quiz takes 3 seconds to load. Is this good UX?
✅ Skeletons (placeholder shapes where content will appear) feel faster than spinners because they set expectations. Users understand 'content is coming' rather than 'something is happening'. Keep skeletons roughly the shape of the final content.
