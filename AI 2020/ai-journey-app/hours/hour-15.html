Part 1 — Concepts (0–15 min)

Making it safe & smart

Real apps meet real edge cases — students asking off-topic questions, submitting empty inputs, or hitting API errors. Build the safety layer that makes your app robust, appropriate, and trustworthy.

🛡️

Input validation

Check inputs before sending to API: empty strings, extremely long inputs, non-text content. Simple JavaScript checks save API costs and improve UX.

🚫

Content filtering

Claude has built-in safety, but you should also add application-level filtering. Redirect off-topic questions gracefully rather than just refusing.

💥

Error handling tiers

Tier 1: Claude returns an error (retry). Tier 2: Claude returns something unparseable (ask to regenerate). Tier 3: Persistent failure (show user-friendly message, log for debugging).

📝

Logging

Log errors with context (what was the prompt? what was the error?) so you can fix issues you'll never reproduce locally. Use a simple console.error() with structured data.

// Comprehensive safety wrapper for your AI calls
async function safeAICall(prompt, options = {}) {
  const { maxRetries = 2, fallback = "I'm having trouble right now. Please try again." } = options;

  // Input validation
  if (!prompt || typeof prompt !== 'string') {
    return { success: false, error: 'Invalid input', response: fallback };
  }

  const trimmed = prompt.trim();
  if (trimmed.length === 0) {
    return { success: false, error: 'Empty input', response: "Please type a question first!" };
  }

  if (trimmed.length > 5000) {
    return { success: false, error: 'Too long', response: "Your question is a bit long — can you shorten it?" };
  }

  // Retry loop
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch("https://api.anthropic.com/v1/messages", {
        method: "POST",
        headers: { "Content-Type": "application/json", "x-api-key": process.env.CLAUDE_API_KEY, "anthropic-version": "2023-06-01" },
        body: JSON.stringify({
          model: "claude-sonnet-4-20250514",
          max_tokens: 1024,
          messages: [{ role: "user", content: trimmed }]
        })
      });

      if (response.status === 429) {
        // Rate limited — wait and retry
        await new Promise(r => setTimeout(r, (attempt + 1) * 2000));
        continue;
      }

      if (!response.ok) {
        throw new Error(\`HTTP \${response.status}\`);
      }

      const data = await response.json();
      return { success: true, response: data.content[0].text };

    } catch (error) {
      console.error('AI call failed', { attempt, error: error.message, promptLength: trimmed.length });

      if (attempt === maxRetries) {
        return { success: false, error: error.message, response: fallback };
      }

      await new Promise(r => setTimeout(r, 1000 * (attempt + 1)));
    }
  }
}

// Topic guard — keep students focused on studying
function isStudyRelated(question) {
  const offTopicPhrases = ['write my essay', 'do my homework', 'what should i eat', 'tell me a joke', 'play a game'];
  const lower = question.toLowerCase();
  return !offTopicPhrases.some(phrase => lower.includes(phrase));
}

async function guardedStudyAnswer(question) {
  if (!isStudyRelated(question)) {
    return "I'm a study helper — I'm best at quizzes, explanations, and helping you understand topics. What subject are you studying today?";
  }
  const result = await safeAICall(question);
  return result.response;
}
Exercise 1Test every failure mode

Systematically test: empty input, 10,000 character input, question with emoji, API key temporarily wrong (change one character), question in an unsupported language. Does your app handle each gracefully?

For each case, your app should: show a helpful specific message (not a generic error), not crash, not expose technical error details to the user. Log the full error details for you (the developer) but show a friendly message to the student.
Exercise 2Write a pre-send checklist

Before any API call is made, run a series of checks. Implement them as a validateInput(text) function that returns {valid: boolean, message: string}.

Checks: (1) not empty, (2) not just whitespace, (3) not too long (>5000 chars), (4) not just numbers (probably not a question), (5) contains at least one letter. Return {valid: false, message: 'Please type a question'} for any failure. Show the message inline next to the input field, not as an alert popup.
Part 3 — Review (45–60 min)

Answer all questions to see your score.

Q1. A student submits an empty question. What should happen?
✅ Client-side validation before the API call. Don't waste API tokens on invalid input. Show an inline message immediately — it's faster, cheaper, and more user-friendly.
Q2. Your app gets a 429 error. What is the right response?
✅ Exponential backoff retry: wait 1s, retry. If 429 again, wait 2s, retry. Then 4s. After 3 failures, show the user a friendly message. Most 429 errors resolve within seconds.
Q3. A student asks 'Write my entire history essay for me.' How should your Study Helper respond?
✅ A friendly redirect is better than a refusal. The student still has a valid study need — the history topic. Redirecting to what the app CAN do keeps them engaged and makes the boundary feel helpful rather than punishing.
